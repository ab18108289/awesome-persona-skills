# 人物 / 角色人格 Skill 合集设计文档

## 1. 项目定位

本仓库的目标是做成 GitHub 上尽可能全的「人物 / 角色人格 Skill 合集」。

仓库只收录具备明确人物感、人设感、角色视角的 Skill 项目，重点覆盖以下两类：

- 真实人物 Skill：热门网红、博主、公众人物、创业者、导师等
- 角色人格 Skill：老板、同事、前任、自己、大师、月老等强人设角色

本仓库不做纯工具合集，不收纯 workflow、纯工程、纯提示词工具类项目。

## 2. 项目目标

### 核心目标

- 让用户第一眼感觉这个仓库真全
- 系统收录 GitHub 上热门人物、网红博主、强人设角色与高需求人格 Skill
- 让用户能按分类快速找到自己想要的人物型 Skill

### 版本目标

- 第一阶段目标不是卡死 100，而是先完成 100+
- 实际执行按「能抓多少抓多少」推进，优先做成 100-150 个的完整收录仓库
- 后续持续扩容，不设固定上限

## 3. 仓库标题与副标题

### 标题

人物 / 角色人格 Skill 合集

### 推荐副标题

系统收录 GitHub 上热门人物、网红博主、强人设角色与高需求人格 Skill，持续更新。

### 可选增强版副标题

尽可能全地收录 GitHub 上热门人物、网红博主、角色人格与人设型 Skill 项目，持续更新。

## 4. 收录边界

### 收录

- 真实人物 IP Skill
- 强角色人格 Skill
- 具备明确使用场景的人设型 Skill
- 有传播性、代表性、实用性或热度的相关项目

### 不收录

- 纯工具型 Skill
- 纯 workflow / engineering skill
- 只是 prompt 杂烩、但没有明确人物或角色定位的项目
- 与人物、人设、角色视角无直接关系的仓库

## 5. 信息架构

README 首页采用分类优先的信息架构，不做单一热榜，不做网站化复杂结构。

推荐顺序如下：

1. 标题 + 一句话定位
2. 项目介绍
3. 快速导航
4. 精选推荐
5. 完整分类清单
6. 新收录
7. 收录标准
8. 投稿方式
9. 免责声明 / 说明

该结构的核心原则是：

- 第一眼建立完整性和价值感
- 用精选区帮助新用户快速理解仓库
- 用完整分类清单证明仓库真的全
- 用新增与投稿机制保证仓库可持续扩容

## 6. README 设计原则

### 总原则

- 分类优先
- 完整性优先
- 长期维护优先
- 传播感为辅，不做纯情绪热榜

### 首页感受目标

用户进入仓库后的第一感受应该是：

- 这个仓库真全
- 这个仓库分类很清楚
- 这个仓库能帮我快速找到想要的人物型 Skill

## 7. 分类体系

推荐采用以下 8 个一级分类：

1. 热门网红 / 博主 Skill
2. 商业人物 / 创业者 Skill
3. 教育 / 职业导师 Skill
4. 情感人物 / 关系人格 Skill
5. 职场角色人格 Skill
6. 玄学人物 / 神秘人格 Skill
7. 经典人格 / 自我镜像 Skill
8. 新收录 / 待观察

### 分类说明

#### 1. 热门网红 / 博主 Skill

收录有传播性、话题性和鲜明个人风格的热门人物 Skill，例如网红、情感博主、现实主义博主、内容型人物等。

#### 2. 商业人物 / 创业者 Skill

收录具备产品观、管理观、商业判断、长期主义视角的人物型 Skill。

#### 3. 教育 / 职业导师 Skill

收录偏教育规划、职业成长、现实建议、方法论指导类人物 Skill。

#### 4. 情感人物 / 关系人格 Skill

收录偏恋爱、关系复盘、两性沟通、情绪陪伴和关系视角类 Skill。

#### 5. 职场角色人格 Skill

收录老板、同事、导师、顾问等工作场景中的强角色人格 Skill。

#### 6. 玄学人物 / 神秘人格 Skill

收录算命、姻缘、命理、大师、传统文化神秘人格等相关 Skill。

#### 7. 经典人格 / 自我镜像 Skill

收录自己、镜像人格、抽象高维人格、神话人格设定等偏人格表达的 Skill。

#### 8. 新收录 / 待观察

收录刚出现、值得纳入但尚需观察热度和代表性的项目。

## 8. 精选区策略

精选区保留，但体量必须克制，避免喧宾夺主。

### 原则

- 精选区只做入口，不替代完整分类
- 精选区总量控制在 12-18 个
- 不做单一热榜
- 按小专题分组展示

### 推荐分组

- 最火人物 Skill
- 最有意思的角色人格 Skill
- 近期新增

精选区的作用是帮助用户快速理解仓库内容，而不是抢走分类主体。

## 9. 单条项目格式

每个条目建议使用统一格式，保证后续扩容可维护。

推荐结构：

- 名称
  一句话介绍
  GitHub: 链接
  标签：若干标签 | Stars: 数字

### 描述要求

每条简介尽量回答两个问题：

- 它模拟的是谁 / 什么人设
- 它最适合解决什么问题

### 标签建议

可使用少量稳定标签，例如：

- 人物
- 角色
- 热门
- 国内
- 海外
- 情感
- 职场
- 玄学
- 创业
- 博主
- 导师

## 10. 数量策略

本仓库不以刚好 100 为目标，而以 100+ 为最低阶段性门槛。

推荐执行方式：

- 核心精选：20-30 个
- 完整收录：100+ 起步
- 实际抓取：能抓多少抓多少，优先保证主题一致和价值感

## 11. 现有样本归类原则

目前已有的样本已经验证这条边界成立：

- 真实人物可收：如户晨风、童锦程、张雪峰、张一鸣
- 强角色人格可收：如老板、同事、前任、自己
- 强设定人格可收：如女娲、大师、月老
- 内容型人设可收：如博主、导师

工具型项目不进入主榜，即使热度高，也不作为本仓库的核心收录对象。

## 12. 收录标准

建议在 README 中明确以下标准：

- 收录 GitHub 上公开可访问的人物 / 人设 / 角色视角 Skill 项目
- 优先收录主题明确、定位清晰、实际可用、具备代表性的仓库
- 兼顾热度、趣味性、实用性和传播性
- 不收纯工具型、纯工程型、纯 workflow 型项目

## 13. 投稿与维护

建议后续补充投稿说明，让仓库具备持续生长能力。

投稿信息建议至少包含：

- Skill 名称
- 人物 / 角色类型
- 一句话介绍
- GitHub 仓库链接
- 推荐理由

## 14. 后续执行顺序

在该设计通过后，推荐按以下顺序进入实现：

1. 先创建 README 骨架
2. 先把已有项目归入正确分类
3. 再批量扩充到 100+
4. 补充 stars、标签、简介统一化
5. 增加投稿与收录标准
